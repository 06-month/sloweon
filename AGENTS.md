# AGENTS.md

## 역할

이 저장소의 구현 작업자는 남성 컨템포러리 패션 쇼핑몰을 구현하는 소프트웨어 엔지니어다. 구현 전에는 반드시 기획 문서와 요구사항 문서를 읽고, 작업 계획과 진행 기록을 문서화해야 한다.

## 제품 기준 문서

구현 전 아래 문서를 먼저 확인한다.

- `docs/shopping-mall-pyd.md`
- `docs/shopping-mall-planning.md`
- `docs/customer-personas.md`
- `docs/information-architecture.md`
- `docs/user-scenarios.md`

문서와 실제 구현이 충돌하면 임의로 구현하지 말고, `plan.md`에 가정과 결정 필요 사항을 기록한 뒤 가장 보수적인 범위로 진행한다.

## 참고 문서

필독 대상은 아니지만 홈/PDP/내비게이션 등 비주얼 디자인을 판단할 때 참고한다.

- `docs/design-instruct.md`
- `menswear_demo_assets/README.md`: 가상 브랜드 SLOWEON 기준 상품/이미지 데모 자산(40 SKU). 색상/소재/핏 방향은 `menswear_demo_assets/docs/brand_direction.md`, 상품 데이터는 `menswear_demo_assets/docs/product_catalog.md` 참고

## 페르소나 서브 에이전트

디자인, 개발, QA, 런칭 단계에서 고객 관점 검토가 필요하면 아래 페르소나 서브 에이전트 정의를 사용한다.

- `.agents/persona-kim-dohyun.md`: 스타일 입문 남성
- `.agents/persona-park-junseo.md`: 데일리 직장인
- `.agents/persona-lee-taeo.md`: 트렌드 얼리어답터
- `.agents/persona-choi-minjae.md`: 사이즈 민감 재구매 고객

주요 화면 또는 주요 기능을 완료했다고 판단하기 전 최소 3개 페르소나 관점의 피드백을 확인한다. 런칭 판단 시에는 4개 페르소나 전원의 차단 이슈가 없어야 한다.

실제 서브 에이전트를 실행할 때는 해당 `.agents/persona-*.md` 파일의 내용을 초기 지시문으로 사용한다. 에이전트 실행 결과는 `process.md`에 요약하고, 릴리즈 차단 이슈는 `plan.md`의 남은 작업 또는 체크리스트에 반영한다.

## 필수 작업 문서

구현 작업을 시작하기 전에 루트에 아래 파일이 있어야 한다.

- `plan.md`
- `process.md`

두 파일이 없으면 구현을 시작하기 전에 먼저 생성한다. 이미 있으면 현재 작업에 맞게 갱신한다.

## plan.md 작성 규칙

`plan.md`는 구현 전 계획 문서다. 구현자는 코드를 수정하기 전에 다음 항목을 채운다.

- 작업 목표
- 구현 범위
- 제외 범위
- 관련 요구사항 ID
- 주요 화면 또는 모듈
- 데이터 모델 또는 상태 구조
- 구현 단계별 체크리스트
- 검증 방법
- 리스크와 대응
- 미정 사항

작업 중 범위가 바뀌면 먼저 `plan.md`를 수정하고, 변경 이유를 `process.md`에 기록한다.

## process.md 작성 규칙

`process.md`는 구현 진행 기록이다. 구현자는 작업 중 계속 갱신한다.

반드시 기록할 내용:

- 작업 시작 시각과 작업 목표
- 읽은 문서와 확인한 요구사항
- 구현한 변경 사항
- 수정한 파일
- 주요 의사결정과 이유
- 실행한 명령어와 결과 요약
- 테스트/검증 결과
- 발견한 문제와 처리 상태
- 남은 작업

진행 기록은 최신 항목이 위로 오도록 작성한다. 단순한 생각 메모가 아니라, 다음 작업자가 이어받을 수 있는 수준으로 구체적으로 적는다.

## 구현 진행 원칙

- 구현은 `plan.md`의 체크리스트 순서를 따른다.
- 체크리스트를 완료할 때마다 `process.md`에 결과를 기록한다.
- 계획과 다르게 구현해야 하면 먼저 `plan.md`를 갱신하고 이유를 남긴다.
- 요구사항이 애매하면 `docs/shopping-mall-pyd.md`의 P0 요구사항을 우선한다.
- 패션몰 특화 기능인 색상/사이즈 옵션, 실측 사이즈표, 모델 착용 정보, 룩북, 코디 세트, 재입고 알림은 일반 쇼핑몰 기능보다 제품 차별화 요소로 취급한다.
- 엔티티 status 등 상태값은 `docs/shopping-mall-pyd.md` 18.1(기타 핵심 상태값 정의)을 따른다. 정의에 없는 상태값이 필요하면 임의로 만들지 말고 `plan.md`에 기록한 뒤 PYD를 갱신한다.

## 품질 기준

- 모바일 우선 반응형 UI를 기본으로 한다.
- 상품 이미지, 상품명, 가격, 색상, 사이즈, 핏 정보가 구매 흐름에서 누락되지 않게 한다.
- IA와 유저 시나리오의 핵심 경로가 실제 라우팅과 화면 구조에 반영되게 한다.
- 고객 페르소나의 주요 문제와 구매 기준이 홈, PLP, PDP, 장바구니, 주문서, 마이페이지에 반영되게 한다.
- 주문, 결제, 재고 차감은 데이터 정합성을 최우선으로 한다.
- 관리자 기능은 권한과 감사 로그를 고려한다.
- 텍스트가 버튼, 카드, 옵션 선택 영역 밖으로 넘치지 않도록 한다.
- 상품 상세에는 사이즈 판단에 필요한 정보가 명확히 보여야 한다.

## 검증 기준

구현 후 가능한 범위에서 다음을 확인한다.

- 빌드 또는 타입 검사가 통과하는지
- 핵심 화면이 모바일과 데스크톱에서 깨지지 않는지
- 상품 목록, 상품 상세, 장바구니, 주문서의 주요 흐름이 동작하는지
- 품절 옵션과 재입고 알림 흐름이 의도대로 표시되는지
- 주문/결제/재고 관련 로직이 중복 처리되지 않는지

검증을 실행하지 못했다면 이유를 `process.md`와 최종 보고에 모두 남긴다.

## 파일 변경 원칙

- 기존 사용자 변경을 임의로 되돌리지 않는다.
- 불필요한 리팩터링을 하지 않는다.
- 문서와 구현이 함께 바뀌면 문서 변경도 같은 작업 기록에 남긴다.
- 새 기능을 추가할 때는 관련 요구사항 ID를 `plan.md`와 `process.md`에 연결한다.
