## 1. 시나리오 작성 기준

- 고객 시나리오는 상품 탐색, 사이즈 판단, 장바구니, 주문, 결제, 배송, 교환/반품을 중심으로 작성한다.
- 관리자 시나리오는 상품 등록, 재고 관리, 주문 처리, CS 처리를 중심으로 작성한다.
- 각 시나리오는 프론트엔드 화면, 백엔드 처리, 데이터베이스 변경, 예외 처리를 함께 검증할 수 있어야 한다.
- 고객 시나리오는 [고객 페르소나](./customer-personas.md)의 이름과 연결해 디자인/개발/QA/런칭 피드백 기준으로 사용한다.

| S-007 재입고 알림 | 이태오 | 품절 옵션의 재입고 신청과 알림이 빠르고 정확한가 |

| S-009 반품/환불 | 최민재, 박준서 | 정책과 환불 상태가 신뢰 가능하게 보이는가 |
| S-010 상품/옵션 등록 | 최민재, 이태오 | 실측/옵션/드롭 정보가 고객 판단에 충분한가 |
| S-011 신규 주문 출고 | 박준서 | 배송 상태와 알림이 신뢰를 주는가 |
| S-012 재고 조정/재입고 알림 | 이태오 | 재입고 수요와 알림 발송이 정확히 운영되는가 |

## 2. 고객 시나리오

### S-001. 신규 고객이 회원가입 후 첫 구매를 한다

관련 요구사항: C-001~C-010, P-001~P-008, PD-001~PD-010, O-001~O-008

사전 조건:

- 고객은 비회원 상태다.
- 구매 가능한 상품과 옵션 재고가 존재한다.

흐름:

1. 고객이 홈에 접속한다.
2. 신상품 또는 카테고리에서 상품 목록으로 이동한다.
3. 상품 상세에서 이미지, 가격, 실측표, 모델 착용 정보를 확인한다.
4. 색상과 사이즈를 선택한다.
5. 장바구니에 담는다.
6. 주문서 진입 중 회원가입을 선택한다.
7. 이메일, 비밀번호, 이름, 휴대폰 번호, 약관 동의를 입력한다.
8. 회원가입 완료 후 로그인 상태가 된다.
9. 배송지를 입력하고 결제수단을 선택한다.
10. 결제를 완료한다.
11. 주문 완료 화면에서 주문번호를 확인한다.

성공 기준:

- 사용자 계정이 생성된다.
- 장바구니 상품이 주문서에 유지된다.
- 결제 성공 시 주문과 주문 상품이 생성된다.
- 재고가 정확히 차감된다.
- 마이페이지에서 주문을 확인할 수 있다.

예외:

- 이메일 중복이면 가입을 차단하고 오류를 표시한다.
- 결제 직전 재고가 부족하면 결제를 차단하고 장바구니로 안내한다.
- 비밀번호 정책을 만족하지 않으면 가입을 차단한다.

## 3. 인증 시나리오

### S-002. 회원이 로그인하고 로그아웃한다

관련 요구사항: C-005~C-009, FE-002, BE-003, BE-004

흐름:

1. 회원이 로그인 화면에 진입한다.
2. 이메일과 비밀번호를 입력한다.
3. 서버가 계정 상태와 비밀번호를 검증한다.
4. 로그인 성공 시 세션 또는 토큰이 발급된다.
5. 회원이 마이페이지에 접근한다.
6. 회원이 로그아웃을 선택한다.
7. 서버가 현재 세션 또는 리프레시 토큰을 폐기한다.
8. 사용자가 마이페이지에 다시 접근하면 로그인 화면으로 이동한다.

성공 기준:

- 로그인 성공 후 인증 필요한 화면에 접근할 수 있다.
- 로그아웃 후 기존 인증 정보로 보호 API를 호출할 수 없다.
- 인증 상태가 프론트엔드에 즉시 반영된다.

예외:

- 비밀번호가 틀리면 실패 횟수를 기록한다.
- 잠긴 계정은 로그인할 수 없다.
- 만료된 세션은 자동 갱신 또는 재로그인을 요구한다.

### S-003. 회원이 비밀번호를 재설정한다

관련 요구사항: C-004, C-010, BE-004, BE-005

흐름:

1. 회원이 비밀번호 재설정 화면에 진입한다.
2. 이메일 또는 휴대폰 번호를 입력한다.
3. 서버가 인증 토큰을 생성하고 알림을 발송한다.
4. 회원이 인증 링크 또는 코드를 입력한다.
5. 새 비밀번호를 설정한다.
6. 서버가 비밀번호를 해시로 저장하고 기존 세션을 무효화한다.
7. 회원이 새 비밀번호로 로그인한다.

성공 기준:

- 인증 토큰은 만료 시간이 있다.
- 사용된 토큰은 재사용할 수 없다.
- 기존 로그인 세션은 폐기된다.

## 4. 상품 탐색/사이즈 시나리오

### S-004. 사이즈에 민감한 고객이 실측표와 착용 리뷰를 보고 구매한다

관련 요구사항: PD-004, PD-005, PD-009, R-002

흐름:

1. 고객이 와이드 팬츠 카테고리에 진입한다.
2. 사이즈, 핏, 색상 필터를 적용한다.
3. 상품 상세에서 모델 키, 몸무게, 착용 사이즈를 확인한다.
4. 실측 사이즈표를 확인한다.
5. 자신과 비슷한 체형의 착용 리뷰를 확인한다.
6. 적절한 색상과 사이즈를 선택한다.
7. 장바구니에 담는다.

성공 기준:

- 상품 상세에서 실측표와 모델 착용 정보가 누락되지 않는다.
- 착용 리뷰는 구매 사이즈와 핏 평가를 보여준다.
- 선택한 옵션의 재고 상태가 정확히 표시된다.

## 5. 룩북/코디 시나리오

### S-005. 스타일 입문자가 룩북에서 코디 세트를 구매한다

관련 요구사항: P-007, P-008, PD-008, O-002

흐름:

1. 고객이 홈에서 룩북을 선택한다.
2. 룩북 상세에서 착장 이미지와 스타일 태그를 확인한다.
3. 착용 상품 목록을 확인한다.
4. 상의, 하의, 아우터 각각의 색상/사이즈를 선택한다.
5. 코디 세트를 장바구니에 한 번에 담는다.
6. 품절 상품이 있으면 재입고 알림 또는 대체 상품을 확인한다.
7. 주문서에서 결제를 완료한다.

성공 기준:

- 코디 세트의 각 상품이 개별 장바구니 항목으로 생성된다.
- 옵션이 선택되지 않은 상품은 담기 전에 선택을 요구한다.
- 품절 옵션은 장바구니에 담을 수 없다.

## 6. 비회원 구매 시나리오

### S-006. 비회원이 상품을 구매하고 주문을 조회한다

관련 요구사항: C-013, C-014, O-004, O-008

흐름:

1. 비회원 고객이 상품을 장바구니에 담는다.
2. 주문서에서 비회원 주문자 정보와 배송지를 입력한다.
3. 개인정보 수집/이용에 동의한다.
4. 결제를 완료한다.
5. 주문번호와 조회용 인증 정보를 받는다.
6. 비회원 주문 조회 화면에서 주문번호와 인증 정보를 입력한다.
7. 주문 상태와 배송 정보를 확인한다.

성공 기준:

- 비회원도 장바구니와 주문을 완료할 수 있다.
- 비회원 주문 조회는 해당 주문에만 접근 가능하다.
- 비회원 개인정보는 회원 계정과 분리 관리된다.

## 7. 재입고 알림 시나리오

### S-007. 품절 사이즈를 재입고 알림 신청한다

관련 요구사항: C-018, M-004, P-006

흐름:

1. 고객이 상품 상세에서 품절된 색상/사이즈를 선택한다.
2. 구매 버튼 대신 재입고 알림 버튼을 확인한다.
3. 회원은 즉시 신청하고, 비회원은 이메일 또는 휴대폰 정보를 입력한다.
4. 서버가 variant 단위로 재입고 신청을 저장한다.
5. 관리자가 해당 옵션의 재고를 추가한다.
6. 시스템이 신청자에게 재입고 알림을 발송한다.

성공 기준:

- 재입고 신청은 productVariant 단위로 저장된다.
- 동일 사용자의 중복 신청은 정책에 따라 제한된다.
- 알림 발송 이력이 남는다.

## 8. 교환/반품 시나리오

### S-008. 고객이 사이즈 교환을 요청한다

관련 요구사항: O-009, D-005, D-006

흐름:

1. 고객이 배송 완료된 주문 상세에 진입한다.
2. 교환 요청 버튼을 선택한다.
3. 교환 사유로 사이즈 작음 또는 사이즈 큼을 선택한다.
4. 원하는 교환 사이즈를 선택한다.
5. 시스템이 교환 가능 기간과 교환 대상 사이즈 재고를 확인한다.
6. 고객이 요청을 제출한다.
7. 관리자가 요청을 승인한다.
8. 회수/재배송 상태가 주문 상세에 표시된다.

성공 기준:

- 교환 가능 기간이 지나면 요청할 수 없다.
- 교환 대상 옵션 재고가 없으면 요청 제한 또는 대기 상태로 처리한다.
- 교환 요청 상태 이력이 남는다.

### S-009. 고객이 반품/환불을 요청한다

관련 요구사항: O-010, D-006, BE-006, DB-006

흐름:

1. 고객이 주문 상세에서 반품 요청을 선택한다.
2. 반품 사유와 상세 내용을 입력한다.
3. 수거지 정보를 확인한다.
4. 관리자가 반품 요청을 승인한다.
5. 반품 입고 후 환불을 처리한다.
6. 결제 취소, 포인트 복구/회수, 쿠폰 정책이 반영된다.

성공 기준:

- 반품 사유가 저장된다.
- 환불 처리는 결제, 포인트, 쿠폰, 주문 상태와 정합성을 유지한다.
- 주문 상태 이력이 남는다.

## 9. 관리자 시나리오

### S-010. 상품관리자가 상품과 옵션을 등록한다

관련 요구사항: A-002~A-007, DB-001~DB-004

흐름:

1. 상품관리자가 관리자 로그인에 성공한다.
2. 상품 등록 화면에 진입한다.
3. 상품명, 카테고리, 가격, 설명, 스타일 태그를 입력한다.
4. 대표 이미지, 착용컷, 상세 이미지를 등록한다.
5. 색상/사이즈 조합별 옵션과 SKU, 재고를 등록한다.
6. 실측 사이즈표를 입력한다.
7. 모델 착용 정보를 입력한다.
8. 상품 상태를 판매중으로 변경한다.

성공 기준:

- 상품, 이미지, variant, sizeSpec, modelFit 데이터가 생성된다.
- 고객 상품 상세에 모든 정보가 표시된다.
- 관리자 활동 로그가 남는다.

### S-011. 주문관리자가 신규 주문을 출고 처리한다

관련 요구사항: A-008, A-009, D-003, D-004

흐름:

1. 주문관리자가 관리자 대시보드에서 배송 준비 주문을 확인한다.
2. 주문 상세에서 결제 상태와 상품 옵션을 확인한다.
3. 출고 가능하면 주문 상태를 배송 준비로 유지하거나 출고 처리한다.
4. 택배사와 송장번호를 입력한다.
5. 주문 상태가 배송 중으로 변경된다.
6. 고객에게 배송 시작 알림이 발송된다.

성공 기준:

- 송장번호가 주문 상세와 고객 화면에 표시된다.
- 배송 상태 변경 이력이 저장된다.
- 알림 발송 결과가 기록된다.

### S-012. 상품관리자가 재고를 조정하고 재입고 알림을 발송한다

관련 요구사항: A-003, A-011, M-004, DB-006

흐름:

1. 상품관리자가 품절 옵션의 재입고 알림 신청 수를 확인한다.
2. 옵션 재고를 추가한다.
3. 재고 원장이 생성된다.
4. 시스템이 재입고 알림 대상자를 조회한다.
5. 알림을 발송한다.
6. 재입고 신청 상태가 알림 완료로 변경된다.

성공 기준:

- 재고 변경 전후 수량이 원장에 남는다.
- 알림은 품절 옵션 신청자에게만 발송된다.
- 중복 알림 발송을 방지한다.

## 10. 검증 우선순위

1. 인증: 회원가입, 로그인, 로그아웃, 비밀번호 재설정, 보호 라우트
2. 상품: 목록, 상세, 옵션, 실측표, 모델 착용 정보
3. 장바구니: 회원/비회원 담기, 수정, 삭제, 병합
4. 주문/결제: 주문서 계산, 결제 성공/실패, 재고 차감
5. 운영: 상품 등록, 재고 조정, 주문 출고, 교환/반품 처리
6. 패션 특화: 룩북, 코디 세트, 재입고 알림, 착용 리뷰
