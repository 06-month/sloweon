## 1. 기획 개요

### 1.1 서비스 정의

남성 컨템포러리 패션 쇼핑몰은 절제된 미니멀 무드와 편안한 릴랙스 테일러링을 결합한 남성 의류 전문몰이다. 고객은 데일리룩, 출근룩, 원마일웨어, 데이트룩 등 상황과 취향에 맞는 상품을 탐색하고, 핏과 사이즈 정보를 확인한 뒤 구매할 수 있다.

### 1.2 기획 목적

일반 쇼핑몰이 아닌 남성 패션 구매 문제에 집중한 서비스를 만든다. 고객이 가장 자주 겪는 문제인 사이즈 불확실성, 코디 어려움, 상품 비교 피로, 품절 옵션 확인, 교환/반품 불안감을 줄이는 것이 핵심이다.

### 1.3 콘셉트 정의

`컨템포러리`는 절제된 미니멀 무드와 편안한 릴랙스 테일러링을 결합한 스타일이다.

- 실루엣: 릴랙스핏, 세미오버, 박시, 드롭숄더, 와이드 스트레이트, 원턱/투턱, 이지팬츠
- 분위기: 미니멀, 차분함, 시티보이, 원마일웨어, 절제된 프리미엄
- 대표 상품: 린넨/코튼 셔츠, 오픈카라 셔츠, 니트 폴로, 박시 티셔츠, 언스트럭처드 블레이저, 와이드 트라우저, 플리츠 슬랙스, 치노 팬츠, 이지 팬츠, 버뮤다 쇼츠, 셋업
- 구매 기준: 핏, 색감, 사이즈, 코디 활용도, 소재감, 가격

## 2. 브랜드 방향

### 2.1 브랜드 포지션

합리적인 가격대의 남성 컨템포러리 패션 편집형 쇼핑몰로 포지셔닝한다. 과장된 그래픽이나 하이프보다 일상에서 실제로 입기 좋은 미니멀하고 절제된 상품을 중심으로 구성한다.

### 2.2 브랜드 톤

- 절제된 시각 톤
- 상품과 착장 중심의 화면
- 과장된 문구보다 핏, 소재, 실측, 코디 정보를 우선하는 설명
- 빠른 구매보다 확신 있는 구매를 돕는 정보 구조

### 2.3 핵심 메시지

- 오늘 입기 좋은 남성 컨템포러리 스타일
- 핏과 사이즈를 확인하고 실패 없이 구매
- 단품보다 착장으로 보는 쇼핑 경험

## 3. 타깃 사용자

### 3.1 타깃 사용자 유형

| 사용자 | 연령/상황 | 문제 | 제공 가치 |
|---|---|---|---|
| 스타일 입문자 | 20대 초중반 | 어떤 옷을 같이 입어야 할지 모름 | 룩북, 코디 세트, 기본템 추천 |
| 데일리 구매자 | 20대 후반~30대 | 출근/일상에 입을 깔끔한 옷 필요 | 미니멀 컨템포러리 상품, 빠른 탐색 |
| 트렌드 얼리어답터 | 20대 | 세미오버·릴랙스 실루엣, 신상품 드롭 선호 | 드롭, 신상품, 스타일 태그 |
| 사이즈 민감 고객 | 전 연령 | 온라인 구매 시 사이즈 실패 경험 | 실측표, 모델 착용, 착용 리뷰 |
| 재구매 고객 | 기존 구매자 | 마음에 든 핏의 상품을 다시 사고 싶음 | 찜, 재입고 알림, 개인 사이즈 저장 |

### 3.2 고객 페르소나

구체적인 고객 페르소나는 [고객 페르소나 문서](./customer-personas.md)를 기준으로 한다. 이후 디자인, 개발, QA, 런칭 판단 시 아래 4명의 관점으로 피드백을 받는다.

| 이름 | 대표 고객군 | 런칭 전 반드시 확인할 질문 |
|---|---|---|
| 김도현 | 스타일 입문 남성 | 처음 방문해도 무엇을 사야 할지 알 수 있는가 |
| 박준서 | 데일리 직장인 | 상품 정보와 배송/교환 정책을 빠르게 신뢰할 수 있는가 |
| 이태오 | 트렌드 얼리어답터 | 신상품, 룩북, 품절/재입고 정보가 충분히 매력적인가 |
| 최민재 | 사이즈 민감 재구매 고객 | 실측과 착용 리뷰만으로 사이즈 확신을 얻을 수 있는가 |

## 4. 핵심 문제와 해결 방향

| 문제 | 해결 방향 |
|---|---|
| 남성 고객은 코디 조합에 피로를 느낀다. | 룩북과 코디 세트 중심으로 상품을 노출한다. |
| 온라인 의류 구매는 사이즈 실패 가능성이 높다. | 실측표, 모델 착용 정보, 구매자 착용 리뷰를 강화한다. |
| 인기 상품은 품절 옵션이 많다. | 옵션별 품절 표시와 재입고 알림을 제공한다. |
| 상품 목록에서 차이를 파악하기 어렵다. | 핏, 색상, 스타일 태그, 시즌 태그 필터를 제공한다. |
| 교환/반품 조건을 늦게 알게 된다. | 상세와 주문서에서 조건을 명확히 안내한다. |

## 5. 서비스 목표

### 5.1 정성 목표

- 고객이 자신에게 맞는 핏과 사이즈를 더 쉽게 판단하게 한다.
- 단품 탐색뿐 아니라 착장 단위 구매 경험을 제공한다.
- 모바일에서 빠르게 상품을 보고 옵션을 선택할 수 있게 한다.
- 관리자가 상품, 룩북, 재고, 주문을 안정적으로 운영하게 한다.

### 5.2 정량 목표

| 지표 | 1차 목표 |
|---|---:|
| 상품 상세 진입률 | 목록 방문자 기준 40% 이상 |
| 사이즈표 조회율 | 상세 방문자 기준 35% 이상 |
| 장바구니 담기율 | 상세 방문자 기준 8% 이상 |
| 코디 세트 담기율 | 룩북 방문자 기준 5% 이상 |
| 결제 전환율 | 주문서 진입자 기준 60% 이상 |
| 사이즈 사유 반품률 | 전체 주문의 12% 이하 |
| 재입고 알림 신청 전환율 | 품절 옵션 조회자 기준 10% 이상 |
| 모바일 구매 완료 비중 | 전체 주문의 60% 이상 대응 |

## 7. 핵심 화면 기획

### 7.1 홈

홈은 브랜드 콘셉트와 현재 구매할 이유를 빠르게 보여준다.

필수 구성:

- 시즌 메인 비주얼
- 신상품 드롭
- 인기 상품
- 카테고리 바로가기
- 룩북 또는 오늘의 코디
- 베이직 아이템 추천
- 재입고 인기 상품

기획 기준:

- 첫 화면에서 남성 컨템포러리 패션몰임을 즉시 알 수 있어야 한다.
- 모바일에서는 메인 비주얼 다음에 바로 상품 탐색 영역을 보여준다.
- 마케팅 문구보다 상품 이미지와 착장 이미지를 우선한다.

### 7.2 상품 목록

상품 목록은 비교와 탐색의 화면이다.

필수 구성:

- 상품 이미지
- 상품명
- 판매가/할인율
- 색상 스와치
- 핏 태그
- 리뷰 수 또는 별점
- 품절/재입고 상태
- 필터와 정렬

핵심 필터:

- 카테고리
- 가격대
- 색상
- 사이즈
- 핏
- 스타일
- 시즌
- 배송 조건
- 할인 여부

### 7.3 상품 상세

상품 상세는 구매 확신을 만드는 화면이다.

필수 구성:

- 대표 이미지, 착용컷, 디테일컷
- 상품명, 가격, 할인율
- 색상/사이즈 옵션
- 실측 사이즈표
- 모델 착용 정보
- 핏 가이드
- 소재/두께/신축성/비침/세탁 정보
- 함께 입은 상품
- 착용 리뷰
- 교환/반품 안내
- 장바구니/바로구매 버튼

기획 기준:

- 사이즈표와 모델 착용 정보는 숨겨진 부가 정보가 아니라 구매 판단의 핵심 정보로 배치한다.
- 모바일에서 옵션 선택과 구매 버튼은 항상 쉽게 접근 가능해야 한다.
- 품절 사이즈는 선택 불가 상태와 재입고 알림 버튼을 함께 제공한다.

### 7.4 룩북

룩북은 단품이 아닌 착장 단위의 탐색 화면이다.

필수 구성:

- 착장 이미지
- 스타일 태그
- 착용 상품 목록
- 각 상품 가격
- 코디 전체 장바구니 담기
- 품절 상품 대체 안내

기획 기준:

- 고객이 코디를 그대로 따라 살 수 있어야 한다.
- 품절 상품이 있어도 비슷한 대체 상품을 제안할 수 있는 구조를 둔다.

### 7.5 장바구니

필수 구성:

- 상품 이미지
- 색상/사이즈/수량
- 옵션 변경
- 품절/재고 부족 안내
- 선택 삭제
- 선택 주문
- 예상 결제 금액

기획 기준:

- 같은 상품이라도 색상/사이즈가 다르면 별도 항목으로 관리한다.
- 재고가 부족해진 상품은 주문 전에 명확히 안내한다.

### 7.6 주문서

필수 구성:

- 주문 상품 요약
- 배송지
- 배송 요청사항
- 쿠폰
- 포인트
- 결제 수단
- 최종 결제 금액
- 교환/반품 주요 안내
- 약관 동의

기획 기준:

- 최종 결제 금액은 서버 계산값과 일치해야 한다.
- 배송비와 교환/반품 배송비 정책을 명확히 보여준다.

### 7.7 마이페이지

필수 구성:

- 최근 주문
- 배송 상태
- 교환/반품 요청
- 재입고 알림 내역
- 찜한 상품/룩북
- 쿠폰/포인트
- 내 사이즈 정보
- 리뷰 작성 가능 상품
- 문의 내역

기획 기준:

- 패션몰 특성상 내 사이즈 정보와 재입고 알림을 핵심 메뉴로 둔다.
- 주문 상태별 가능한 행동을 분명히 보여준다.

### 7.8 관리자 대시보드

필수 구성:

- 오늘 주문 수/매출
- 배송 준비 필요 주문
- 교환/반품 요청
- 품절 임박 옵션
- 재입고 알림 신청 많은 상품
- 미답변 문의
- 인기 상품/저조 상품
- 사이즈별 판매량

기획 기준:

- 운영자가 당장 처리해야 하는 업무를 먼저 보여준다.
- 패션몰 운영에 중요한 사이즈/재고/반품 지표를 기본 제공한다.

## 8. 주요 사용자 흐름

### 8.1 룩북 기반 구매 흐름

1. 고객이 홈에서 룩북 영역을 선택한다.
2. 착장 이미지와 스타일 태그를 확인한다.
3. 착용 상품 목록에서 원하는 상품을 선택한다.
4. 각 상품의 색상/사이즈를 선택한다.
5. 코디 세트를 장바구니에 담는다.
6. 주문서에서 배송지와 결제수단을 선택한다.
7. 결제를 완료한다.

### 8.2 사이즈 확인 중심 구매 흐름

1. 고객이 상품 상세에 진입한다.
2. 모델 착용 정보와 실측 사이즈표를 확인한다.
3. 구매자 착용 리뷰에서 비슷한 체형의 리뷰를 확인한다.
4. 색상과 사이즈를 선택한다.
5. 장바구니 또는 바로구매를 진행한다.

### 8.3 품절 옵션 재입고 흐름

1. 고객이 품절된 색상/사이즈 옵션을 선택한다.
2. 재입고 알림 버튼을 누른다.
3. 로그인 또는 휴대폰/이메일 정보를 확인한다.
4. 재입고 시 알림을 받는다.
5. 알림 링크로 상품 상세에 재진입해 구매한다.

### 8.4 사이즈 교환 흐름

1. 고객이 배송 완료된 주문 상세에 진입한다.
2. 교환 요청을 선택한다.
3. 교환 사유와 원하는 사이즈를 선택한다.
4. 시스템이 교환 가능 기간과 재고를 확인한다.
5. 관리자가 요청을 승인한다.
6. 회수 및 재배송 상태를 고객에게 표시한다.

## 9. 상품 운영 정책

### 9.2 옵션 정책

- 재입고 예정일이 있는 경우 고객에게 표시할 수 있다.

### 9.3 사이즈 정책

- 의류 상품은 실측 사이즈표 등록을 원칙으로 한다.

- 상품군별 필수 실측 항목을 다르게 둔다.

### 9.4 이미지 정책

- 대표 이미지
- 착용 이미지
- 컬러별 이미지

- 소재 클로즈업 이미지

모든 상품은 최소 대표 이미지 1장과 상세 이미지 1장을 가져야 하며, 1차 출시 기준으로 핵심 상품은 착용컷을 포함한다.

## 11. 운영 정책

### 11.1 주문 운영

- 결제 완료 주문만 출고 대상으로 본다.
- 송장번호 등록 전까지는 배송 준비 상태로 관리한다.
- 주문 상태 변경 이력은 보존한다.

### 11.2 교환/반품 운영

- 사이즈 교환, 단순변심, 상품 불량을 구분한다.
- 교환 요청 시 교환 대상 옵션의 재고를 확인한다.
- 반품 사유는 상품 개선과 사이즈 안내 개선에 활용한다.

### 11.3 CS 운영

- 사이즈 문의는 상품 상세의 실측표와 모델 정보를 기준으로 답변한다.
- 재입고 문의는 관리자 재입고 요청 현황과 연결한다.
- 비밀 문의는 작성자와 관리자만 확인할 수 있다.

## 12. 기술 방향

### 12.1 프론트엔드

- 모바일 우선 반응형 웹으로 구현한다.
- 상품 이미지는 빠른 로딩을 위해 최적화와 지연 로딩을 적용한다.
- 상품 목록 필터, 옵션 선택, 장바구니 금액 변경은 즉시 반영한다.
- 패션몰 특성상 이미지 비율과 텍스트 잘림을 엄격히 관리한다.
- 고객 영역과 관리자 영역의 라우팅, 레이아웃, 권한 가드를 분리한다.
- 로그인 사용자, 비회원 사용자, 관리자 사용자의 상태를 명확히 구분한다.
- 상품/장바구니/주문/회원 정보는 서버 상태로 관리하고 변경 후 캐시 무효화 기준을 둔다.
- 회원가입, 로그인, 주문서, 상품 등록 등 주요 폼은 클라이언트 검증과 서버 검증 결과를 함께 표시한다.

### 12.2 백엔드

- 상품, 옵션, 재고, 주문, 결제, 배송, 교환/반품 도메인을 분리한다.
- 결제와 재고 차감은 트랜잭션 정합성을 우선한다.
- 관리자 변경 사항은 감사 로그로 저장한다.
- 인증, 회원, 상품, 장바구니, 주문, 결제, 배송, CS, 관리자 API를 명확히 분리한다.
- 모든 API는 입력 검증, 권한 검증, 표준 오류 응답을 적용한다.
- 주문 생성, 결제 승인, 재고 차감, 쿠폰/포인트 사용은 하나의 일관된 처리 흐름으로 묶는다.
- 결제 승인, 결제 웹훅, 재고 차감은 멱등성을 보장한다.

### 12.3 데이터베이스

- 관계형 데이터베이스를 기준으로 회원, 상품, 옵션, 주문, 결제, 배송, 재고, 쿠폰, 포인트, 리뷰, 문의를 모델링한다.
- 스키마 변경은 마이그레이션으로 관리한다.
- 이메일, 주문번호, 결제 키, SKU, 색상/사이즈 옵션 조합 등 중복되면 안 되는 값에는 고유 제약을 둔다.
- 주문 내역은 상품명, 색상, 사이즈, 가격, 배송지 정보를 주문 시점 스냅샷으로 저장한다.
- 재고, 포인트, 결제, 환불은 원장성 이력을 남긴다.
- 상품 검색, 카테고리, 주문번호, 회원 주문 목록, 재고 조회에는 인덱스를 둔다.
- 탈퇴 회원과 개인정보 보관 정책을 고려해 마스킹 또는 분리 보관 기준을 둔다.

### 12.4 인증/보안

- 비밀번호는 평문 저장을 금지하고 안전한 해시로 저장한다.
- 로그인, 로그아웃, 세션 만료, 토큰 갱신, 비밀번호 재설정 흐름을 모두 구현 대상으로 본다.
- 관리자 기능은 일반 회원과 별도 권한 정책을 적용한다.
- 반복 로그인 실패, 비정상 요청, 권한 없는 관리자 API 접근을 차단하고 로그로 남긴다.
- 관리자 화면의 개인정보는 필요한 범위에서만 표시하고 민감 정보는 마스킹한다.

### 12.5 외부 연동

- PG 결제
- SMS/알림톡/이메일
- 택배사 배송조회
- 주소 검색

- 분석 도구

## 15. 출시 전략

### 15.1 1차 출시 목표

1차 출시는 남성 컨템포러리 패션몰의 기본 구매 흐름과 패션 특화 정보 제공을 완성하는 것을 목표로 한다. 룩북, 실측 사이즈표, 모델 착용 정보, 재입고 알림은 일반 쇼핑몰과 구분되는 핵심 기능으로 우선 포함한다.

### 15.2 베타 운영

- 초기 상품 30~50개로 상품 등록 테스트
- 핵심 카테고리: 상의, 하의, 아우터, 셋업
- 룩북 5~10개 구성
- 실제 PG 테스트 결제
- 사이즈 교환/반품 운영 테스트
- 모바일 브라우저별 QA

- P0 요구사항 구현 완료
- 주요 상품 실측 사이즈표 등록 완료
- 대표 상품 착용 이미지 등록 완료
- 결제 성공/실패/취소 검증 완료
- 재고 차감/복구 검증 완료
- 관리자 주문/배송 처리 검증 완료
